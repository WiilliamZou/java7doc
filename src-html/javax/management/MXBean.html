054/**
055    <p>Annotation to mark an interface explicitly as being an MXBean
056    interface, or as not being an MXBean interface.  By default, an
057    interface is an MXBean interface if its name ends with {@code
058    MXBean}, as in {@code SomethingMXBean}.  The following interfaces
059    are MXBean interfaces:</p>
060
061    <pre>
062    public interface WhatsitMXBean {}
063
064    &#64;MXBean
065    public interface Whatsit1Interface {}
066
067    &#64;MXBean(true)
068    public interface Whatsit2Interface {}
069    </pre>
070
071    <p>The following interfaces are not MXBean interfaces:</p>
072
073    <pre>
074    public interface Whatsit3Interface{}
075
076    &#64;MXBean(false)
077    public interface MisleadingMXBean {}
078    </pre>
079
080    <h3 id="MXBean-spec">MXBean specification</a></h3>
081
082    <p>The MXBean concept provides a simple way to code an MBean
083      that only references a predefined set of types, the ones defined
084      by {@link javax.management.openmbean}.  In this way, you can be
085      sure that your MBean will be usable by any client, including
086      remote clients, without any requirement that the client have
087      access to <em>model-specific classes</em> representing the types
088      of your MBeans.</p>
089
090    <p>The concepts are easier to understand by comparison with the
091      Standard MBean concept.  Here is how a managed object might be
092      represented as a Standard MBean, and as an MXBean:</p>
093
094    <table border="1" cellpadding="5">
095      <tr>
096        <th>Standard MBean</th><th>MXBean</th>
097      </tr>
098      <tr>
099        <td><pre>
100public interface MemoryPool<b>MBean</b> {
101    String getName();
102    MemoryUsage getUsage();
103    // ...
104}
105          </pre></td>
106        <td><pre>
107public interface MemoryPool<b>MXBean</b> {
108    String getName();
109    MemoryUsage getUsage();
110    // ...
111}
112          </pre></td>
113      </tr>
114    </table>
115
116    <p>As you can see, the definitions are very similar.  The only
117      difference is that the convention for naming the interface is to use
118      <code><em>Something</em>MXBean</code> for MXBeans, rather than
119      <code><em>Something</em>MBean</code> for Standard MBeans.</p>
120
121    <p>In this managed object, there is an attribute called
122      <code>Usage</code> of type {@link MemoryUsage}.  The point of an
123      attribute like this is that it gives a coherent snapshot of a set
124      of data items.  For example, it might include the current amount
125      of used memory in the memory pool, and the current maximum of the
126      memory pool.  If these were separate items, obtained with separate
127      {@link MBeanServer#getAttribute getAttribute} calls, then we could
128      get values seen at different times that were not consistent.  We
129      might get a <code>used</code> value that was greater than the
130      <code>max</code> value.</p>
131
132    <p>So, we might define <code>MemoryUsage</code> like this:</p>
133
134    <table border="1" cellpadding="5">
135      <tr>
136        <th>Standard MBean</th><th>MXBean</th>
137      </tr>
138      <tr>
139        <td><pre>
140public class MemoryUsage <b>implements Serializable</b> {
141    // standard JavaBean conventions with getters
142
143    public MemoryUsage(long init, long used,
144                       long committed, long max) {...}
145    long getInit() {...}
146    long getUsed() {...}
147    long getCommitted() {...}
148    long getMax() {...}
149}
150          </pre></td>
151        <td><pre>
152public class MemoryUsage {
153    // standard JavaBean conventions with getters
154    <b>&#64;ConstructorProperties({"init", "used", "committed", "max"})</b>
155    public MemoryUsage(long init, long used,
156                       long committed, long max) {...}
157    long getInit() {...}
158    long getUsed() {...}
159    long getCommitted() {...}
160    long getMax() {...}
161}
162          </pre></td>
163      </tr>
164    </table>
165
166    <p>The definitions are the same in the two cases, except
167      that with the MXBean, <code>MemoryUsage</code> no longer needs to
168      be marked <code>Serializable</code> (though it can be).  On
169      the other hand, we have added a {@code @ConstructorProperties} annotation
170      to link the constructor parameters to the corresponding getters.
171      We will see more about this below.</p>
172
173    <p><code>MemoryUsage</code> is a <em>model-specific class</em>.
174      With Standard MBeans, a client of the MBean Server cannot access the
175      <code>Usage</code> attribute if it does not know the class
176      <code>MemoryUsage</code>.  Suppose the client is a generic console
177      based on JMX technology.  Then the console would have to be
178      configured with the model-specific classes of every application it
179      might connect to.  The problem is even worse for clients that are
180      not written in the Java language.  Then there may not be any way
181      to tell the client what a <code>MemoryUsage</code> looks like.</p>
182
183    <p>This is where MXBeans differ from Standard MBeans.  Although we
184      define the management interface in almost exactly the same way,
185      the MXBean framework <em>converts</em> model-specific classes into
186      standard classes from the Java platform.  Using arrays and the
187      {@link javax.management.openmbean.CompositeData CompositeData} and
188      {@link javax.management.openmbean.TabularData TabularData} classes
189      from the standard {@link javax.management.openmbean} package, it
190      is possible to build data structures of arbitrary complexity
191      using only standard classes.</p>
192
193    <p>This becomes clearer if we compare what the clients of the two
194      models might look like:</p>
195
196    <table border="1" cellpadding="5">
197      <tr>
198        <th>Standard MBean</th><th>MXBean</th>
199      </tr>
200      <tr>
201        <td><pre>
202String name = (String)
203    mbeanServer.{@link MBeanServer#getAttribute
204    getAttribute}(objectName, "Name");
205<b>MemoryUsage</b> usage = (<b>MemoryUsage</b>)
206    mbeanServer.getAttribute(objectName, "Usage");
207<b>long used = usage.getUsed();</b>
208          </pre></td>
209        <td><pre>
210String name = (String)
211    mbeanServer.{@link MBeanServer#getAttribute
212    getAttribute}(objectName, "Name");
213<b>{@link CompositeData}</b> usage = (<b>CompositeData</b>)
214    mbeanServer.getAttribute(objectName, "Usage");
215<b>long used = (Long) usage.{@link CompositeData#get get}("used");</b>
216          </pre></td>
217    </table>
218
219    <p>For attributes with simple types like <code>String</code>, the
220      code is the same.  But for attributes with complex types, the
221      Standard MBean code requires the client to know the model-specific
222      class <code>MemoryUsage</code>, while the MXBean code requires no
223      non-standard classes.</p>
224
225    <p>The client code shown here is slightly more complicated for the
226      MXBean client.  But, if the client does in fact know the model,
227      here the interface <code>MemoryPoolMXBean</code> and the
228      class <code>MemoryUsage</code>, then it can construct a
229      <em>proxy</em>.  This is the recommended way to interact with
230      managed objects when you know the model beforehand, regardless
231      of whether you are using Standard MBeans or MXBeans:</p>
232
233    <table border="1" cellpadding="5">
234      <tr>
235        <th>Standard MBean</th><th>MXBean</th>
236      </tr>
237      <tr>
238        <td><pre>
239MemoryPool<b>MBean</b> proxy =
240    JMX.<b>{@link JMX#newMBeanProxy(MBeanServerConnection, ObjectName,
241              Class) newMBeanProxy}</b>(
242        mbeanServer,
243        objectName,
244        MemoryPool<b>MBean</b>.class);
245String name = proxy.getName();
246MemoryUsage usage = proxy.getUsage();
247long used = usage.getUsed();
248          </pre></td>
249        <td><pre>
250MemoryPool<b>MXBean</b> proxy =
251    JMX.<b>{@link JMX#newMXBeanProxy(MBeanServerConnection, ObjectName,
252              Class) newMXBeanProxy}</b>(
253        mbeanServer,
254        objectName,
255        MemoryPool<b>MXBean</b>.class);
256String name = proxy.getName();
257MemoryUsage usage = proxy.getUsage();
258long used = usage.getUsed();
259          </pre></td>
260      </tr>
261    </table>
262
263    <p>Implementing the MemoryPool object works similarly for both
264      Standard MBeans and MXBeans.</p>
265
266    <table border="1" cellpadding="5">
267      <tr>
268        <th>Standard MBean</th><th>MXBean</th>
269      </tr>
270      <tr>
271        <td><pre>
272public class MemoryPool
273        implements MemoryPool<b>MBean</b> {
274    public String getName() {...}
275    public MemoryUsage getUsage() {...}
276    // ...
277}
278          </pre></td>
279        <td><pre>
280public class MemoryPool
281        implements MemoryPool<b>MXBean</b> {
282    public String getName() {...}
283    public MemoryUsage getUsage() {...}
284    // ...
285}
286          </pre></td>
287      </tr>
288    </table>
289
290    <p>Registering the MBean in the MBean Server works in the same way
291      in both cases:</p>
292
293    <table border="1" cellpadding="5">

316    <h2 id="mxbean-def">Definition of an MXBean</h2>
317
318    <p>An MXBean is a kind of MBean.  An MXBean object can be
319      registered directly in the MBean Server, or it can be used as an
320      argument to {@link StandardMBean} and the resultant MBean
321      registered in the MBean Server.</p>
322
323    <p>When an object is registered in the MBean Server using the
324      {@code registerMBean} or {@code createMBean} methods of the
325      {@link MBeanServer} interface, the object's class is examined
326      to determine what type of MBean it is:</p>
327
328    <ul>
329      <li>If the class implements the interface {@link DynamicMBean}
330        then the MBean is a Dynamic MBean.  Note that the class
331        {@code StandardMBean} implements this interface, so this
332        case applies to a Standard MBean or MXBean created using
333        the class {@code StandardMBean}.</li>
334
335      <li>Otherwise, if the class matches the Standard MBean naming
336        conventions, then the MBean is a Standard MBean.</li>
337
338      <li>Otherwise, it may be an MXBean.  The set of interfaces
339        implemented by the object is examined for interfaces that:
340
341        <ul>
342          <li>have a class name <code><em>S</em>MXBean</code> where
343            <code><em>S</em></code> is any non-empty string, and
344            do not have an annotation {@code @MXBean(false)}; and/or</li>
345          <li>have an annotation {@code @MXBean(true)}
346            or just {@code @MXBean}.</li>
347        </ul>
348
349        If there is exactly one such interface, or if there is one
350        such interface that is a subinterface of all the others, then
351        the object is an MXBean.  The interface in question is the
352        <em>MXBean interface</em>.  In the example above, the MXBean
353        interface is {@code MemoryPoolMXBean}.
354
355      <li>If none of these conditions is met, the MBean is invalid and
356        the attempt to register it will generate {@link
357        NotCompliantMBeanException}.
358    </ul>
359
360    <p>Every Java type that appears as the parameter or return type of a
361      method in an MXBean interface must be <em>convertible</em> using
362      the rules below.  Additionally, parameters must be
363      <em>reconstructible</em> as defined below.</p>
364
365    <p>An attempt to construct an MXBean that does not conform to the
366      above rules will produce an exception.</p>
367
368
369    <h2 id="naming-conv">Naming conventions</h2>
370
371    <p>The same naming conventions are applied to the methods in an
372      MXBean as in a Standard MBean:</p>
373
374    <ol>
375      <li>A method <code><em>T</em> get<em>N</em>()</code>, where
376        <code><em>T</em></code> is a Java type (not <code>void</code>)
377        and <code><em>N</em></code> is a non-empty string, specifies
378        that there is a readable attribute called
379        <code><em>N</em></code>.  The Java type and Open type of the
380        attribute are determined by the mapping rules below.
381        The method {@code final Class getClass()} inherited from {@code
382        Object} is ignored when looking for getters.</li>
383
384      <li>A method <code>boolean is<em>N</em>()</code> specifies that
385        there is a readable attribute called <code><em>N</em></code>
386        with Java type <code>boolean</code> and Open type
387        <code>SimpleType.Boolean</code>.</li>
388
389      <li>A method <code>void set<em>N</em>(<em>T</em> x)</code>
390        specifies that there is a writeable attribute called
391        <code><em>N</em></code>.  The Java type and Open type of the
392        attribute are determined by the mapping rules below.  (Of
393        course, the name <code>x</code> of the parameter is
394        irrelevant.)</li>
395
396      <li>Every other method specifies that there is an operation with
397        the same name as the method.  The Java type and Open type of the
398        return value and of each parameter are determined by the mapping
399        rules below.</li>
400    </ol>
401
402    <p>The rules for <code>get<em>N</em></code> and
403      <code>is<em>N</em></code> collectively define the notion of a
404      <em>getter</em>.  The rule for <code>set<em>N</em></code> defines
405      the notion of a <em>setter</em>.</p>
406
407    <p>It is an error for there to be two getters with the same name, or
408      two setters with the same name.  If there is a getter and a setter
409      for the same name, then the type <code><em>T</em></code> in both
410      must be the same.  In this case the attribute is read/write.  If
411      there is only a getter or only a setter, the attribute is
412      read-only or write-only respectively.</p>
413

415    <h2 id="mapping-rules">Type mapping rules</h2>
416
417    <p>An MXBean is a kind of Open MBean, as defined by the {@link
418      javax.management.openmbean} package.  This means that the types of
419      attributes, operation parameters, and operation return values must
420      all be describable using <em>Open Types</em>, that is the four
421      standard subclasses of {@link javax.management.openmbean.OpenType}.
422      MXBeans achieve this by mapping Java types into Open Types.</p>
423
424    <p>For every Java type <em>J</em>, the MXBean mapping is described
425      by the following information:</p>
426
427    <ul>
428      <li>The corresponding Open Type, <em>opentype(J)</em>.  This is
429        an instance of a subclass of {@link
430        javax.management.openmbean.OpenType}.</li>
431      <li>The <em>mapped</em> Java type, <em>opendata(J)</em>, which is
432        always the same for any given <em>opentype(J)</em>.  This is a Java
433        class.</li>
434      <li>How a value is converted from type <em>J</em> to type
435        <em>opendata(J)</em>.</li>
436      <li>How a value is converted from type <em>opendata(J)</em> to
437        type <em>J</em>, if it can be.</li>
438    </ul>
439
440    <p>For example, for the Java type {@code List<String>}:</p>
441
442    <ul>
443      <li>The Open Type, <em>opentype(</em>{@code
444        List<String>}<em>)</em>, is {@link ArrayType}<code>(1, </code>{@link
445          SimpleType#STRING}<code>)</code>, representing a 1-dimensional
446          array of <code>String</code>s.</li>
447      <li>The mapped Java type, <em>opendata(</em>{@code
448        List<String>}<em>)</em>, is {@code String[]}.</li>
449      <li>A {@code List<String>} can be converted to a {@code String[]}
450          using {@link List#toArray(Object[]) List.toArray(new
451          String[0])}.</li>
452      <li>A {@code String[]} can be converted to a {@code List<String>}
453          using {@link Arrays#asList Arrays.asList}.</li>
454    </ul>
455
456    <p>If no mapping rules exist to derive <em>opentype(J)</em> from
457      <em>J</em>, then <em>J</em> cannot be the type of a method
458      parameter or return value in an MXBean interface.</p>
459
460    <p id="reconstructible-def">If there is a way to convert
461      <em>opendata(J)</em> back to <em>J</em> then we say that <em>J</em> is
462      <em>reconstructible</em>.  All method parameters in an MXBean
463      interface must be reconstructible, because when the MXBean
464      framework is invoking a method it will need to convert those
465      parameters from <em>opendata(J)</em> to <em>J</em>.  In a proxy
466      generated by {@link JMX#newMXBeanProxy(MBeanServerConnection,
467      ObjectName, Class) JMX.newMXBeanProxy}, it is the return values
468      of the methods in the MXBean interface that must be
469      reconstructible.</p>
470
471    <p>Null values are allowed for all Java types and Open Types,
472      except primitive Java types where they are not possible.  When
473      converting from type <em>J</em> to type <em>opendata(J)</em> or
474      from type <em>opendata(J)</em> to type <em>J</em>, a null value is
475      mapped to a null value.</p>
476

554    <h3>Mappings for primitive types</h3>
555
556    <p>The 8 primitive Java types
557      ({@code boolean}, {@code byte}, {@code short}, {@code int}, {@code
558      long}, {@code float}, {@code double}, {@code char}) are mapped to the
559      corresponding boxed types from {@code java.lang}, namely {@code
560      Boolean}, {@code Byte}, etc.  The Open Type is the corresponding
561      {@code SimpleType}.  Thus, <em>opentype(</em>{@code
562      long}<em>)</em> is {@code SimpleType.LONG}, and
563      <em>opendata(</em>{@code long}<em>)</em> is {@code
564      java.lang.Long}.</p>
565
566    <p>An array of primitive type such as {@code long[]} can be represented
567      directly as an Open Type.  Thus, <em>openType(</em>{@code
568      long[]}<em>)</em> is {@code
569      ArrayType.getPrimitiveArrayType(long[].class)}, and
570      <em>opendata(</em>{@code long[]}<em>)</em> is {@code
571      long[]}.</p>
572
573    <p>In practice, the difference between a plain {@code int} and {@code
574      Integer}, etc, does not show up because operations in the JMX API
575      are always on Java objects, not primitives.  However, the
576      difference <em>does</em> show up with arrays.</p>
577
578
579    <h3>Mappings for collections ({@code List<}<em>E</em>{@code >} etc)</h3>
580
581    <p>A {@code List<}<em>E</em>{@code >} or {@code
582      Set<}<em>E</em>{@code >}, such as {@code List<String>} or {@code
583        Set<ObjectName>}, is mapped in the same way as an array of the
584          same element type, such as {@code String[]} or {@code
585          ObjectName[]}.</p>
586
587    <p>A {@code SortedSet<}<em>E</em>{@code >} is also mapped in the
588      same way as an <em>E</em>{@code []}, but it is only convertible if
589      <em>E</em> is a class or interface that implements {@link
590      java.lang.Comparable}.  Thus, a {@code SortedSet<String>} or
591        {@code SortedSet<Integer>} is convertible, but a {@code
592          SortedSet<int[]>} or {@code SortedSet<List<String>>} is not.  The
593                conversion of a {@code SortedSet} instance will fail with an
594                {@code IllegalArgumentException} if it has a
595                non-null {@link java.util.SortedSet#comparator()
596                comparator()}.</p>
597
598    <p>A {@code List<}<em>E</em>{@code >} is reconstructed as a
599      {@code java.util.ArrayList<}<em>E</em>{@code >};
600      a {@code Set<}<em>E</em>{@code >} as a
601      {@code java.util.HashSet<}<em>E</em>{@code >};
602      a {@code SortedSet<}<em>E</em>{@code >} as a
603      {@code java.util.TreeSet<}<em>E</em>{@code >}.</p>
604
605
606    <h3>Mappings for maps ({@code Map<}<em>K</em>,<em>V</em>{@code >} etc)</h3>
607
608    <p>A {@code Map<}<em>K</em>,<em>V</em>{@code >} or {@code
609      SortedMap<}<em>K</em>,<em>V</em>{@code >}, for example {@code
610      Map<String,ObjectName>}, has Open Type {@link TabularType} and is mapped
611        to a {@link TabularData}.
612        The {@code TabularType} has two items called {@code key} and
613        {@code value}.  The Open Type of {@code key} is
614        <em>opentype(K)</em>, and the Open Type of {@code value} is
615        <em>opentype(V)</em>.  The index of the {@code TabularType} is the
616        single item {@code key}.</p>
617
618    <p>For example, the {@code TabularType} for a {@code
619      Map<String,ObjectName>} might be constructed with code like
620        this:</p>
621
622    <pre>
623String typeName =
624    "java.util.Map&lt;java.lang.String, javax.management.ObjectName&gt;";
625String[] keyValue =
626    new String[] {"key", "value"};
627OpenType[] openTypes =
628    new OpenType[] {SimpleType.STRING, SimpleType.OBJECTNAME};
629CompositeType rowType =
630    new CompositeType(typeName, typeName, keyValue, keyValue, openTypes);
631TabularType tabularType =
632    new TabularType(typeName, typeName, rowType, new String[] {"key"});
633    </pre>
634
635    <p>The {@code typeName} here is determined by the <a href="#type-names">
636      type name rules</a> detailed below.
637
638    <p>A {@code SortedMap<}<em>K</em>,<em>V</em>{@code >} is mapped in the
639      same way, but it is only convertible if
640      <em>K</em> is a class or interface that implements {@link
641      java.lang.Comparable}.  Thus, a {@code SortedMap<String,int[]>}
642        is convertible, but a
643        {@code SortedMap<int[],String>} is not.  The conversion of a
644          {@code SortedMap} instance will fail with an {@code
645          IllegalArgumentException} if it has a non-null {@link
646          java.util.SortedMap#comparator() comparator()}.</p>
647
648    <p>A {@code Map<}<em>K</em>,<em>V</em>{@code >} is reconstructed as
649      a {@code java.util.HashMap<}<em>K</em>,<em>V</em>{@code >};
650      a {@code SortedMap<}<em>K</em>,<em>V</em>{@code >} as
651      a {@code java.util.TreeMap<}<em>K</em>,<em>V</em>{@code >}.</p>
652
653    <p>{@code TabularData} is an interface.  The concrete class that is
654      used to represent a {@code Map<}<em>K</em>,<em>V</em>{@code >} as
655      Open Data is {@link TabularDataSupport},
656      or another class implementing {@code
657      TabularData} that serializes as {@code TabularDataSupport}.</p>
658
659
660    <h3 id="mxbean-map">Mappings for MXBean interfaces</h3>
661
662    <p>An MXBean interface, or a type referenced within an MXBean
663      interface, can reference another MXBean interface, <em>J</em>.
664      Then <em>opentype(J)</em> is {@code SimpleType.OBJECTNAME} and
665      <em>opendata(J)</em> is {@code ObjectName}.</p>
666
667    <p>For example, suppose you have two MXBean interfaces like this:</p>
668
669    <pre>
670public interface ProductMXBean {
671    public ModuleMXBean[] getModules();
672}
673
674public interface ModuleMXBean {
675    public ProductMXBean getProduct();
676}
677    </pre>
678
679    <p>The object implementing the {@code ModuleMXBean} interface
680      returns from its {@code getProduct} method an object
681      implementing the {@code ProductMXBean} interface.  The
682      {@code ModuleMXBean} object and the returned {@code
683      ProductMXBean} objects must both be registered as MXBeans in the
684      same MBean Server.</p>
685
686    <p>The method {@code ModuleMXBean.getProduct()} defines an
687      attribute called {@code Product}.  The Open Type for this
688      attribute is {@code SimpleType.OBJECTNAME}, and the corresponding
689      {@code ObjectName} value will be the name under which the
690      referenced {@code ProductMXBean} is registered in the MBean
691      Server.</p>
692
693    <p>If you make an MXBean proxy for a {@code ModuleMXBean} and
694      call its {@code getProduct()} method, the proxy will map the
695      {@code ObjectName} back into a {@code ProductMXBean} by making
696      another MXBean proxy.  More formally, when a proxy made with
697      {@link JMX#newMXBeanProxy(MBeanServerConnection, ObjectName,
698       Class)
699      JMX.newMXBeanProxy(mbeanServerConnection, objectNameX,
700      interfaceX)} needs to map {@code objectNameY} back into {@code
701      interfaceY}, another MXBean interface, it does so with {@code
702      JMX.newMXBeanProxy(mbeanServerConnection, objectNameY,
703      interfaceY)}.  The implementation may return a proxy that was
704      previously created by a call to {@code JMX.newMXBeanProxy}
705      with the same parameters, or it may create a new proxy.</p>
706
707    <p>The reverse mapping is illustrated by the following change to the
708      {@code ModuleMXBean} interface:</p>
709
710    <pre>
711public interface ModuleMXBean {
712    public ProductMXBean getProduct();
713    public void setProduct(ProductMXBean c);
714}
715    </pre>
716
717    <p>The presence of the {@code setProduct} method now means that the
718      {@code Product} attribute is read/write.  As before, the value
719      of this attribute is an {@code ObjectName}.  When the attribute is
720      set, the {@code ObjectName} must be converted into the
721      {@code ProductMXBean} object that the {@code setProduct} method
722      expects.  This object will be an MXBean proxy for the given
723      {@code ObjectName} in the same MBean Server.</p>
724
725    <p>If you make an MXBean proxy for a {@code ModuleMXBean} and
726      call its {@code setProduct} method, the proxy will map its
727      {@code ProductMXBean} argument back into an {@code ObjectName}.
728      This will only work if the argument is in fact another proxy,
729      for a {@code ProductMXBean} in the same {@code
730      MBeanServerConnection}.  The proxy can have been returned from
731      another proxy (like {@code ModuleMXBean.getProduct()} which
732      returns a proxy for a {@code ProductMXBean}); or it can have
733      been created by {@link
734      JMX#newMXBeanProxy(MBeanServerConnection, ObjectName, Class)
735      JMX.newMXBeanProxy}; or it can have been created using {@link
736      java.lang.reflect.Proxy Proxy} with an invocation handler that
737      is {@link MBeanServerInvocationHandler} or a subclass.</p>
738
739    <p>If the same MXBean were registered under two different
740      {@code ObjectName}s, a reference to that MXBean from another
741      MXBean would be ambiguous.  Therefore, if an MXBean object is
742      already registered in an MBean Server and an attempt is made to
743      register it in the same MBean Server under another name, the
744      result is an {@link InstanceAlreadyExistsException}.  Registering
745      the same MBean object under more than one name is discouraged in
746      general, notably because it does not work well for MBeans that are
747      {@link NotificationBroadcaster}s.</p>
748
749    <h3 id="composite-map">Mappings for other types</h3>
750
751    <p>Given a Java class or interface <em>J</em> that does not match the other
752      rules in the table above, the MXBean framework will attempt to map
753      it to a {@link CompositeType} as follows.  The type name of this
754      {@code CompositeType} is determined by the <a href="#type-names">
755      type name rules</a> below.</p>
756
757    <p>The class is examined for getters using the conventions
758      <a href="#naming-conv">above</a>.  (Getters must be public
759      instance methods.)  If there are no getters, or if
760      any getter has a type that is not convertible, then <em>J</em> is
761      not convertible.</p>
762
763    <p>If there is at least one getter and every getter has a
764      convertible type, then <em>opentype(J)</em> is a {@code
765      CompositeType} with one item for every getter.  If the getter is
766
767    <blockquote>
768      <code><em>T</em> get<em>Name</em>()</code>
769    </blockquote>
770
771    then the item in the {@code CompositeType} is called {@code name}
772    and has type <em>opentype(T)</em>.  For example, if the item is
773
774    <blockquote>
775      <code>String getOwner()</code>
776    </blockquote>
777
778    then the item is called {@code owner} and has Open Type {@code
779    SimpleType.STRING}.  If the getter is
780
781    <blockquote>
782      <code>boolean is<em>Name</em>()</code>
783    </blockquote>
784
785    then the item in the {@code CompositeType} is called {@code name}
786    and has type {@code SimpleType.BOOLEAN}.</p>
787
788    <p>Notice that the first character (or code point) is converted to
789      lower case.  This follows the Java Beans convention, which for
790      historical reasons is different from the Standard MBean
791      convention.  In a Standard MBean or MXBean interface, a method
792      {@code getOwner} defines an attribute called {@code Owner}, while
793      in a Java Bean or mapped {@code CompositeType}, a method {@code
794      getOwner} defines a property or item called {@code owner}.</p>
795
796    <p>If two methods produce the same item name (for example, {@code
797      getOwner} and {@code isOwner}, or {@code getOwner} and {@code
798      getowner}) then the type is not convertible.</p>
799
800    <p>When the Open Type is {@code CompositeType}, the corresponding
801      mapped Java type (<em>opendata(J)</em>) is {@link
802      CompositeData}.  The mapping from an instance of <em>J</em> to a
803      {@code CompositeData} corresponding to the {@code CompositeType}
804      just described is done as follows.  First, if <em>J</em>
805      implements the interface {@link CompositeDataView}, then that
806      interface's {@link CompositeDataView#toCompositeData
807      toCompositeData} method is called to do the conversion.
808      Otherwise, the {@code CompositeData} is constructed by calling
809      the getter for each item and converting it to the corresponding
810      Open Data type.  Thus, a getter such as</p>
811
812    <blockquote>
813      {@code List<String> getNames()}
814    </blockquote>
815
816    <p>will have been mapped to an item with name "{@code names}" and
817      Open Type {@code ArrayType(1, SimpleType.STRING)}.  The conversion
818      to {@code CompositeData} will call {@code getNames()} and convert
819      the resultant {@code List<String>} into a {@code String[]} for the
820        item "{@code names}".</p>
821
822    <p>{@code CompositeData} is an interface.  The concrete class that is
823      used to represent a type as Open Data is {@link
824      CompositeDataSupport}, or another class implementing {@code
825      CompositeData} that serializes as {@code
826      CompositeDataSupport}.</p>
827
828
829    <h4>Reconstructing an instance of Java type <em>J</em> from
830      a {@code CompositeData}</h4>
831
832    <p>If <em>opendata(J)</em> is {@code CompositeData} for a Java type
833      <em>J</em>, then either an instance of <em>J</em> can be
834      reconstructed from a {@code CompositeData}, or <em>J</em> is not
835      reconstructible.  If any item in the {@code CompositeData} is not
836      reconstructible, then <em>J</em> is not reconstructible either.</p>
837
838    <p>For any given <em>J</em>, the following rules are consulted to
839      determine how to reconstruct instances of <em>J</em> from
840      {@code CompositeData}.  The first applicable rule in the list is
841      the one that will be used.</p>
842
843    <ol>
844
845      <li><p>If <em>J</em> has a method<br>
846        {@code public static }<em>J </em>{@code from(CompositeData cd)}<br>
847        then that method is called to reconstruct an instance of
848        <em>J</em>.</p></li>
849
850      <li><p>Otherwise, if <em>J</em> has at least one public
851        constructor with a {@link java.beans.ConstructorProperties
852        ConstructorProperties} annotation, then one
853        of those constructors (not necessarily always the same one)
854        will be called to reconstruct an instance of <em>J</em>.
855        Every such annotation must list as many strings as the
856        constructor has parameters; each string must name a property
857        corresponding to a getter of <em>J</em>; and the type of this
858        getter must be the same as the corresponding constructor
859        parameter.  It is not an error for there to be getters that
860        are not mentioned in the {@code ConstructorProperties} annotation
861        (these may correspond to information that is not needed to
862        reconstruct the object).</p>
863
864        <p>An instance of <em>J</em> is reconstructed by calling a
865        constructor with the appropriate reconstructed items from the
866        {@code CompositeData}.  The constructor to be called will be
867        determined at runtime based on the items actually present in
868        the {@code CompositeData}, given that this {@code
869        CompositeData} might come from an earlier version of
870        <em>J</em> where not all the items were present.  A
871        constructor is <em>applicable</em> if all the properties named
872        in its {@code ConstructorProperties} annotation are present as items
873        in the {@code CompositeData}.  If no constructor is
874        applicable, then the attempt to reconstruct <em>J</em> fails.</p>
875
876        <p>For any possible combination of properties, it must be the
877        case that either (a) there are no applicable constructors, or
878        (b) there is exactly one applicable constructor, or (c) one of
879        the applicable constructors names a proper superset of the
880        properties named by each other applicable constructor.  (In
881        other words, there should never be ambiguity over which
882        constructor to choose.)  If this condition is not true, then
883        <em>J</em> is not reconstructible.</p></li>
884
885      <li><p>Otherwise, if <em>J</em> has a public no-arg constructor, and
886        for every getter in <em>J</em> with type
887        <em>T</em> and name <em>N</em> there is a corresponding setter
888        with the same name and type, then an instance of <em>J</em> is
889        constructed with the no-arg constructor and the setters are
890        called with the reconstructed items from the {@code CompositeData}
891        to restore the values.  For example, if there is a method<br>
892        {@code public List<String> getNames()}<br>
893          then there must also be a method<br>
894          {@code public void setNames(List<String> names)}<br>
895            for this rule to apply.</p>
896
897        <p>If the {@code CompositeData} came from an earlier version of
898        <em>J</em>, some items might not be present.  In this case,
899        the corresponding setters will not be called.</p></li>
900
901      <li><p>Otherwise, if <em>J</em> is an interface that has no methods
902        other than getters, an instance of <em>J</em> is constructed
903        using a {@link java.lang.reflect.Proxy} with a {@link
904        CompositeDataInvocationHandler} backed by the {@code
905        CompositeData} being converted.</p></li>
906
907      <li><p>Otherwise, <em>J</em> is not reconstructible.</p></li>
908    </ol>
909
910    <p>Here are examples showing different ways to code a type {@code
911      NamedNumber} that consists of an {@code int} and a {@code
912      String}.  In each case, the {@code CompositeType} looks like this:</p>
913
914    <blockquote>
915      <pre>
916{@link CompositeType}(
917    "NamedNumber",                      // typeName
918    "NamedNumber",                      // description
919    new String[] {"number", "name"},    // itemNames
920    new String[] {"number", "name"},    // itemDescriptions
921    new OpenType[] {SimpleType.INTEGER,
922                    SimpleType.STRING}  // itemTypes
923);
924      </pre>
925    </blockquote>
926
927    <ol>
928      <li>Static {@code from} method:
929
930        <blockquote>
931          <pre>
932public class NamedNumber {
933    public int getNumber() {return number;}
934    public String getName() {return name;}
935    private NamedNumber(int number, String name) {
936        this.number = number;
937        this.name = name;
938    }
939    <b>public static NamedNumber from(CompositeData cd)</b> {
940        return new NamedNumber((Integer) cd.get("number"),
941                               (String) cd.get("name"));
942    }
943    private final int number;
944    private final String name;
945}
946          </pre>
947        </blockquote>
948      </li>
949
950      <li>Public constructor with <code>&#64;ConstructorProperties</code> annotation:
951
952        <blockquote>
953          <pre>
954public class NamedNumber {
955    public int getNumber() {return number;}
956    public String getName() {return name;}
957    <b>&#64;ConstructorProperties({"number", "name"})
958    public NamedNumber(int number, String name)</b> {
959        this.number = number;
960        this.name = name;
961    }
962    private final int number;
963    private final String name;
964}
965          </pre>
966        </blockquote>
967      </li>
968
969      <li>Setter for every getter:
970
971        <blockquote>
972          <pre>
973public class NamedNumber {
974    public int getNumber() {return number;}
975    public void <b>setNumber</b>(int number) {this.number = number;}
976    public String getName() {return name;}
977    public void <b>setName</b>(String name) {this.name = name;}
978    <b>public NamedNumber()</b> {}
979    private int number;
980    private String name;
981}
982          </pre>
983        </blockquote>
984      </li>
985
986      <li>Interface with only getters:
987
988        <blockquote>
989          <pre>
990public interface NamedNumber {
991    public int getNumber();
992    public String getName();
993}
994          </pre>
995        </blockquote>
996      </li>
997    </ol>
998
999    <p>It is usually better for classes that simply represent a
1000      collection of data to be <em>immutable</em>.  An instance of an
1001      immutable class cannot be changed after it has been constructed.
1002      Notice that {@code CompositeData} itself is immutable.
1003      Immutability has many advantages, notably with regard to
1004      thread-safety and security.  So the approach using setters should
1005      generally be avoided if possible.</p>

1008    <h3>Recursive types</h3>
1009
1010    <p>Recursive (self-referential) types cannot be used in MXBean
1011      interfaces.  This is a consequence of the immutability of {@link
1012      CompositeType}.  For example, the following type could not be the
1013      type of an attribute, because it refers to itself:</p>
1014
1015    <pre>
1016public interface <b>Node</b> {
1017    public String getName();
1018    public int getPriority();
1019    public <b>Node</b> getNext();
1020}
1021</pre>
1022
1023    <p>It is always possible to rewrite recursive types like this so
1024      they are no longer recursive.  Doing so may require introducing
1025      new types.  For example:</p>
1026
1027    <pre>
1028public interface <b>NodeList</b> {
1029    public List&lt;Node&gt; getNodes();
1030}
1031
1032public interface Node {
1033    public String getName();
1034    public int getPriority();
1035}
1036</pre>

1038    <h3>MBeanInfo contents for an MXBean</h3>
1039
1040    <p>An MXBean is a type of Open MBean.  However, for compatibility
1041      reasons, its {@link MBeanInfo} is not an {@link OpenMBeanInfo}.
1042      In particular, when the type of an attribute, parameter, or
1043      operation return value is a primitive type such as {@code int},
1044      or is {@code void} (for a return type), then the attribute,
1045      parameter, or operation will be represented respectively by an
1046      {@link MBeanAttributeInfo}, {@link MBeanParameterInfo}, or
1047      {@link MBeanOperationInfo} whose {@code getType()} or {@code
1048      getReturnType()} returns the primitive name ("{@code int}" etc).
1049      This is so even though the mapping rules above specify that the
1050      <em>opendata</em> mapping is the wrapped type ({@code Integer}
1051      etc).</p>
1052
1053    <p>The array of public constructors returned by {@link
1054      MBeanInfo#getConstructors()} for an MXBean that is directly
1055      registered in the MBean Server will contain all of the public
1056      constructors of that MXBean.  If the class of the MXBean is not
1057      public then its constructors are not considered public either.
1058      The list returned for an MXBean that is constructed using the
1059      {@link StandardMBean} class is derived in the same way as for
1060      Standard MBeans.  Regardless of how the MXBean was constructed,
1061      its constructor parameters are not subject to MXBean mapping
1062      rules and do not have a corresponding {@code OpenType}.</p>
1063
1064    <p>The array of notification types returned by {@link
1065      MBeanInfo#getNotifications()} for an MXBean that is directly
1066      registered in the MBean Server will be empty if the MXBean does
1067      not implement the {@link NotificationBroadcaster} interface.
1068      Otherwise, it will be the result of calling {@link
1069      NotificationBroadcaster#getNotificationInfo()} at the time the MXBean
1070      was registered.  Even if the result of this method changes
1071      subsequently, the result of {@code MBeanInfo.getNotifications()}
1072      will not.  The list returned for an MXBean that is constructed
1073      using the {@link StandardMBean} or {@link StandardEmitterMBean}
1074      class is derived in the same way as for Standard MBeans.</p>
1075
1076    <p>The {@link Descriptor} for all of the
1077      {@code MBeanAttributeInfo}, {@code MBeanParameterInfo}, and
1078      {@code MBeanOperationInfo} objects contained in the {@code MBeanInfo}
1079      will have a field {@code openType} whose value is the {@link OpenType}
1080      specified by the mapping rules above.  So even when {@code getType()}
1081      is "{@code int}", {@code getDescriptor().getField("openType")} will
1082      be {@link SimpleType#INTEGER}.</p>
1083
1084    <p>The {@code Descriptor} for each of these objects will also have a
1085      field {@code originalType} that is a string representing the Java type
1086      that appeared in the MXBean interface.  The format of this string
1087      is described in the section <a href="#type-names">Type Names</a>
1088      below.</p>
1089
1090    <p>The {@code Descriptor} for the {@code MBeanInfo} will have a field
1091      {@code mxbean} whose value is the string "{@code true}".</p>
1092
1093
1094    <h3 id="type-names">Type Names</h3>
1095
1096    <p>Sometimes the unmapped type <em>T</em> of a method parameter or
1097    return value in an MXBean must be represented as a string.  If
1098    <em>T</em> is a non-generic type, this string is the value
1099    returned by {@link Class#getName()}.  Otherwise it is the value of
1100    <em>genericstring(T)</em>, defined as follows:
1101
1102    <ul>
1103
1104      <li>If <em>T</em> is a non-generic non-array type,
1105      <em>genericstring(T)</em> is the value returned by {@link
1106      Class#getName()}, for example {@code "int"} or {@code
1107      "java.lang.String"}.
1108
1109      <li>If <em>T</em> is an array <em>E[]</em>,
1110      <em>genericstring(T)</em> is <em>genericstring(E)</em> followed
1111      by {@code "[]"}.  For example, <em>genericstring({@code int[]})</em>
1112      is {@code "int[]"}, and <em>genericstring({@code
1113      List<String>[][]})</em> is {@code
1114      "java.util.List<java.lang.String>[][]"}.
1115
1116    <li>Otherwise, <em>T</em> is a parameterized type such as {@code
1117    List<String>} and <em>genericstring(T)</em> consists of the
1118    following: the fully-qualified name of the parameterized type as
1119    returned by {@code Class.getName()}; a left angle bracket ({@code
1120    "<"}); <em>genericstring(A)</em> where <em>A</em> is the first
1121    type parameter; if there is a second type parameter <em>B</em>
1122    then {@code ", "} (a comma and a single space) followed by
1123    <em>genericstring(B)</em>; a right angle bracket ({@code ">"}).
1124
1125    </ul>
1126
1127    <p>Note that if a method returns {@code int[]}, this will be
1128      represented by the string {@code "[I"} returned by {@code
1129      Class.getName()}, but if a method returns {@code List<int[]>},
1130      this will be represented by the string {@code
1131      "java.util.List<int[]>"}.
1132
1133    <h3>Exceptions</h3>
1134
1135    <p>A problem with mapping <em>from</em> Java types <em>to</em>
1136      Open types is signaled with an {@link OpenDataException}.  This
1137      can happen when an MXBean interface is being analyzed, for
1138      example if it references a type like {@link java.util.Random
1139      java.util.Random} that has no getters.  Or it can happen when an
1140      instance is being converted (a return value from a method in an
1141      MXBean or a parameter to a method in an MXBean proxy), for
1142      example when converting from {@code SortedSet<String>} to {@code
1143      String[]} if the {@code SortedSet} has a non-null {@code
1144      Comparator}.</p>
1145
1146    <p>A problem with mapping <em>to</em> Java types <em>from</em>
1147      Open types is signaled with an {@link InvalidObjectException}.
1148      This can happen when an MXBean interface is being analyzed, for
1149      example if it references a type that is not
1150      <em>reconstructible</em> according to the rules above, in a
1151      context where a reconstructible type is required.  Or it can
1152      happen when an instance is being converted (a parameter to a
1153      method in an MXBean or a return value from a method in an MXBean
1154      proxy), for example from a String to an Enum if there is no Enum
1155      constant with that name.</p>
1156
1157    <p>Depending on the context, the {@code OpenDataException} or
1158      {@code InvalidObjectException} may be wrapped in another
1159      exception such as {@link RuntimeMBeanException} or {@link
1160      UndeclaredThrowableException}.  For every thrown exception,
1161      the condition <em>C</em> will be true: "<em>e</em> is {@code
1162      OpenDataException} or {@code InvalidObjectException} (as
1163      appropriate), or <em>C</em> is true of <em>e</em>.{@link
1164      Throwable#getCause() getCause()}".</p>
1165
1166   @since 1.6
1167*/

1169@Documented

1171@Target(ElementType.TYPE)
1172public @interface MXBean {
1173    /**
1174       True if the annotated interface is an MXBean interface.
1175       @return true if the annotated interface is an MXBean interface.
1176    */
1177    boolean value() default true;
